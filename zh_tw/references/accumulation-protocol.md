# 累積協定

Deferred reference。於回顧階段、交付階段或失敗升級時載入。

每次執行都向知識空間存入資產。問題不是「有沒有成功？」而是「產出了什麼可重用的東西？」

---

## 架構

```
Accumulation Protocol
├── Failure = Data（與壓力升級共同啟動）
│   └── 每一層級在升級前萃取排除資料
├── Retrospective Pipeline（將內嵌 4 步驟擴展為 5 步驟 + 累積）
│   ├── 步驟 1–3：回顧目標、評估結果、分析原因（內嵌）
│   ├── 步驟 4：萃取模式（結構化 — 類別 + 防範措施）
│   ├── 步驟 5：反事實檢查（決策 vs. 執行教訓）
│   └── 累積：依累積協定存入
├── Accumulation Protocol（存入機制）
│   ├── 存入類型：SOP、排除記錄、已驗證事實、搜尋路徑
│   └── 門檻：可重複（2+）且非瑣碎（3+ 步驟）
├── Persistence Filter（跨 session 存活）
│   └── 反模式 + 專案陷阱 — 必須同時為隱性且持久
└── Information Transparency（交付階段 — 不過濾的決策 context）
```

流程：執行 → 失敗觸發資料萃取 → 回顧正式化 → 協定存入 → 持久化過濾跨 session 存活。

---

## 失敗 = 資料

*與壓力升級共同啟動。升級前先回答：「此次嘗試排除了 _____，並將解空間縮小至 _____。」*

| 層級 | 資料萃取 |
|------|----------|
| L1 | 萃取：L0 的嘗試排除了什麼？ |
| L2 | 萃取：L1 縮小了哪些假說空間？ |
| L3 | 承接：L0–L2 累積的排除地圖 |
| L4 | 完整排除地圖用以界定剩餘解空間 |

產生零資訊的嘗試 — 而非失敗本身 — 才是真正的浪費。

---

## 回顧管線

將 SKILL.md 中內嵌的 4 步驟回顧擴展為 5 步驟 + 累積。

| 步驟 | 動作 | 產出 |
|------|------|------|
| 1–3 | 回顧目標 → 評估結果 → 分析原因 | （與內嵌版本相同） |
| 4 | 萃取模式（結構化） | 類別 + 防範措施 |
| 5 | 反事實檢查 | 決策規則或執行筆記 |
| → | 累積 | 依累積協定存入 |

### 步驟 4 — 結構化萃取

- [發生了什麼] → [為什麼 — 結構性根因，非表面症狀]
- [屬於哪類問題] → [可重用產出：SOP 候選 / 啟發式規則 / 搜尋路徑]
- 「下次小心一點」不算模式。模式必須命名問題類別與對應的防範措施。

### 步驟 5 — 反事實檢查

「如果重來一次，步驟 1 會不同嗎？」

| 回答 | 含義 | 萃取 |
|------|------|------|
| 是 | 教訓在於決策，而非執行 | 決策規則 |
| 否 | 方法正確；代價不可避免，或問題出在執行層面 | 執行筆記 |

連續 3 個以上主要任務零存入 → 檢視萃取是否確實在發生。

## 累積協定

### 存入類型

| 類型 | 範例 | 重用價值 |
|------|------|----------|
| SOP 候選 | 「遇到 [錯誤類別]，先檢查 [X] 再檢查 [Y] — 可省下 [N] 個步驟」 | 可重複的程序 |
| 排除記錄 | 「針對 [問題類型]：已排除 [A, B, C]。剩餘空間：[D, E]」 | 防止重複探索 |
| 已驗證事實 | 「[事實] 已透過 [來源/工具] 確認。與假設 [Z] 矛盾」 | 修正錯誤前提 |
| 搜尋路徑 | 「關於 [主題]：[來源 A] 具權威性、[B] 已過時、[C] 有誤導性」 | 加速未來搜尋 |

### 門檻

存入條件 — 兩者皆須滿足：

| 條件 | 檢驗 |
|------|------|
| 可重複 | 預計出現 2 次以上 |
| 非瑣碎 | 重現需 3 個以上步驟 |

任一不符 → 不存入。過度累積 = 雜訊。

### 流程

1. **執行期間**：可重用發現出現時即刻標記（不要等到回顧階段）
2. **回顧步驟 4–5**：正式存入
3. **過濾**：存入前套用門檻

---

## 持久化過濾

累積協定的存入預設為 session 範圍。跨 session 存活必須通過過濾。

### 存活條件

兩者皆須為真：

| 條件 | 檢驗 |
|------|------|
| 隱性 | 不看記錄就會再踩（從程式碼/文件中看不出來） |
| 持久 | 跨 session 後仍然成立（不是暫時狀態） |

任一不符 → session 範圍的存入即已足夠。

### 持久存入類型

| 類型 | 內容 | 格式 |
|------|------|------|
| 反模式 | 行為層 — 犯過的錯 + 正確做法 | `[日期] [情境] 踩坑：___ → 教訓：___` |
| 專案陷阱 | 環境層 — 專案/工具特有的已知地雷 | `[情境] 陷阱：___ → 正確做法：___` |

### 載入時機

同一專案目錄下啟動 session 時 → 載入作為接收階段的前置 context（與認知簡報平行）。

### 儲存位置

基礎設施決策 — 由使用者的 memory 系統、CLAUDE.md 或外部 state 檔案承載。本協定只定義「存什麼」與「什麼格式」，不綁定儲存位置。

---

## 資訊透明度

*交付階段原則。與交付關卡共同啟動。*

所有與決策相關的資訊一律不經過濾地呈現 — 嘗試過什麼、排除了什麼、殘留哪些風險、哪些假設仍然成立。使用者的下一個決策取決於資訊的完整性。過濾複雜度 = 降低下游決策品質。

---

## 反模式

| 模式 | 問題 |
|------|------|
| 「修好了 bug。」— 未萃取 | 一次性消費。零累積。修復隨 session 結束而消失。 |
| 「試了 X，沒用。」— 未記錄排除了什麼 | 違反「失敗 = 資料」原則。下次嘗試可能重新探索 X。 |
| 每個兩行的修復都套用 SOP 流程 | 過度累積的雜訊。門檻的存在是有原因的。 |
| 存入模糊的教訓（「設定檔要小心」） | 不可執行。有效的存入必須命名具體的類別、觸發條件與防範措施。 |
