# Verification Gates

品質把關的關卡規格。每個關卡具備觸發條件、操作流程、通過/失敗判定標準。

Deferred reference — 於 Execute、Deliver 或 L3+ 時載入。

---

## 7-Item Checklist（L3+ 強制執行）

每一項都需要行內證據 — 不是打勾，是舉證。

```
- [ ] 逐字讀過失敗訊號了嗎？
      └─ 關鍵訊號: ___ 解讀: ___

- [ ] 用工具搜尋過核心問題了嗎？
      └─ 搜尋內容: ___ 結果摘要（一行）: ___

- [ ] 讀過失敗位置周圍的原始上下文了嗎？
      └─ 已讀取的 File:line: ___ 相關發現: ___

- [ ] 所有假設都用工具驗證過了嗎？
      └─ 假設: ___ 驗證指令: ___ 結果: ___

- [ ] 嘗試過完全相反的假說了嗎？
      └─ 原始假說: ___ 反轉假說: ___ 結果: ___

- [ ] 能在最小範圍內重現嗎？
      └─ 最小重現步驟: ___ 輸出: ___

- [ ] 換過工具/方法/角度/技術棧了嗎？
      └─ 先前方案: ___ 改用方案: ___ 理由: ___
```

## Blue Army Self-Check

**觸發條件：** 實作任何方案之前。

**操作流程：**
1. 用 30 秒扮演自己的對手
2. 最可能在哪裡出問題？
3. 邊界案例？異常輸入？
4. 不可見路徑覆蓋率：錯誤處理路徑、未測試的程式碼分支、使用者不會直接觸發的邊界案例

沒人看的地方被攻破，就會導致所有人看的地方被攻破。

**通過：** 至少辨識出一個潛在失敗點並加以處理。
**失敗：** 未經自我攻擊就直接實作 — 在審查時標記。

#### 交付後韌性測試（混沌工程變體）
- **觸發條件**：Deliver 階段之後，針對複雜或高風險的輸出
- **操作流程**：刻意對已交付的方案注入邊界案例、畸形輸入、邊界條件。測試：能否優雅地處理，還是靜默地崩潰？
- **通過**：輸出以可預期的方式降級，錯誤訊息具資訊性，無靜默損壞
- **失敗**：靜默失敗、未定義行為、或壓力下出現資料損壞 → 重新開啟，修復韌性缺口後才能最終交付

## Reward Hack Test

**觸發條件：** 宣告問題已解決之前。

**操作流程：** 這是解決了「通用問題」，還是只是通過了當前的測試條件？
- 修復對特定案例和通用案例都有效？
- 換一組輸入是否會暴露同樣的底層問題？
- 是補丁還是真正的修復？

**通過：** 修復已針對通用案例驗證。
**失敗：** 僅為補丁 — 繼續依循方法論，不得宣告完成。

## End-to-End Control

**觸發條件：** 任務依賴外部假設 — API 行為、環境設定、上游輸出、錯誤歸因。

**操作流程：** 關鍵路徑不得依賴未經驗證的外部假設。

| 面向 | 依賴式（脆弱） | 端對端（受控） |
|-----------|--------------------|-----------------------|
| API 行為 | 憑記憶假設 | 閱讀文件或實測呼叫 |
| 使用者環境 | 猜測 OS/版本/設定 | 以程式偵測或直接詢問 |
| 上游輸出 | 未經驗證就信任 | 使用前先驗證輸入 |
| 錯誤原因 | 歸咎於環境 | 先用工具驗證（Red Line 2） |

**通過：** 關鍵路徑上的所有假設均已用工具驗證。
**失敗：** 關鍵路徑中存在未驗證的假設 — 在繼續之前標記為外部依賴。

## Keeper Test

**觸發條件：** 對失敗或停滯的輸出進行迭代之前 — 決定繼續或捨棄。

**操作流程：** 「如果這個輸出即將被替換，我會爭取保留它嗎？」
- 會 — 繼續迭代，這個輸出有值得延續的價值
- 不會 — 捨棄並以結構性不同的方法重新開始

沒有改善觀察期。立刻決定。

**通過：** 輸出值得保留 — 迭代。
**失敗：** 不值得保留 — 捨棄。不要對不值得保留的輸出投入漸進式改善。

## Grayscale Deployment

**觸發條件：** 執行變更時。Execute 階段。

**操作流程：** 漸進式套用變更：
1. 套用到最小可行範圍（單一函式、單一模組）
2. 在小範圍內測試並驗證
3. 從測試結果收集回饋
4. 視需要調整
5. 驗證調整結果
6. 擴展到完整範圍

**通過：** 小範圍驗證成功 — 允許擴展。
**失敗：** 小範圍失敗 — 先在小範圍修復。未經小範圍確認，絕不擴展到全範圍。

## End-User Perspective

**觸發條件：** 交付之前。Deliver 階段。

**操作流程：** 切換視角 — 一個對實作一無所知的終端使用者收到這份輸出。
- 他們能否在沒有實作背景的情況下直接使用？
- 檢查到介面層級：函式簽名、錯誤訊息、回傳格式、變數命名

**通過：** 零實作知識的人可以立即使用該輸出。
**失敗：** 需要內部知識才能理解 — 交付前先簡化介面。

## Difficulty Reporting Format

**觸發條件：** T6 啟動 — 困難已被承認，進度繼續推進。

**操作流程：**
1. **以協助為目的** — 回報是為了推進任務，不是為了產生藉口
2. **可行動** — 附上具體的替代方案或下一步行動

**通過：** 回報包含明確的下一步行動。
**失敗：** 「這很有挑戰性」卻沒有具體下一步 = 逃避，不是回報。
